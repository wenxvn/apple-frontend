# Agent 工作约定

本文件是本仓库每个新任务的入口。在阅读业务实现文件前，必须按以下顺序读取。

## 读取顺序

1. 阅读 `docs/ARCHITECTURE.md`，理解项目边界并定位相关区域。
2. 阅读 `docs/STATE.md`，恢复当前状态、近期决策和下一步。
3. 仅在需要某项历史决策、工具操作或修改细节时，定向查询 `docs/PROJECT_LOG.md`；不要默认完整阅读。
4. 仅在延续先前会话且该文件存在时，阅读 `memory.md`。
5. 处理 UI 工作时，若 `ui-registry.md` 存在，也必须先阅读。

不要默认递归读取整个仓库或逐个打开所有文件。应根据架构文档和当前任务选择最少的相关文件；仅当文档无法定位问题时，才使用定向的 `rg` 搜索。

## 必须遵循的流程

1. 对非简单功能或架构变更，在编辑前使用 `skills/architect/SKILL.md` 对齐术语、关键决策和实施计划。
2. 只实现已确认且在范围内的变更，保留与当前任务无关的用户修改。
3. 运行范围最小但足够的验证；完成特性后，使用 `skills/review/SKILL.md` 按计划和架构进行审查。
4. 每次修改 UI 组件后，使用 `skills/imprint/SKILL.md`，将可复用的视觉模式更新到 `ui-registry.md`。
5. 工作失败或思路混乱时，先使用 `skills/recover/SKILL.md` 判断失败类型，再尝试下一次修复。
6. 每个工作会话结束时，使用 `skills/remember/SKILL.md` 将精简且不含敏感信息的续作上下文保存到 `memory.md`。

## 变更记录

同一任务中的每次实质变更都要更新 `docs/STATE.md` 和 `docs/PROJECT_LOG.md`：修改的文件、目的、验证结果、使用的外部工具或服务、决策和剩余工作。`STATE.md` 始终保持精简且只反映最新状态；`PROJECT_LOG.md` 只能追加新条目，不得改写历史记录。不得记录凭据、令牌、私有 URL 或其他敏感信息。

下列情况属于实质变更，必须记录：代码、依赖、配置、架构边界、部署、Git 提交或推送、外部服务状态，以及会影响后续工作的重要决策。纯阅读、无改动排查、格式检查和没有结论的探索不需要单独写入记录。

## 工具记录

使用外部工具或服务时（例如 GitHub、Vercel、npm、浏览器自动化或 API），要在 `docs/PROJECT_LOG.md` 中记录用途、结果和不敏感的项目或资源名称。部署与项目初始化必须分开记录。

## 工程保护规则

- 优先选择最小、可逆的改动；会扩大范围的假设必须明确说明。
- 暂存或提交前先检查 `git status`，不得纳入无关文件。
- 不提交密钥、生成的依赖目录、构建产物或本地服务元数据。
- 架构和记录文档应保持精简、最新；它们是后续任务节省 token 的事实来源。
- 每项任务完成前，对照 `docs/TASK_TEMPLATE.md` 的收尾清单确认实现、验证、记录和版本状态均已完成。
